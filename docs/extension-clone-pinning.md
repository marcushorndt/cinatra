# Pinned extension clone-back (CI) ‚Äî locks, canary, and the bump recipe

## Why

Merge-gating CI materializes every `cinatraDevExtensions` companion repo
(79 today) into `extensions/<scope>/<name>/` before `pnpm install
--frozen-lockfile`. Those checkouts are pnpm workspace members, so when the
clone-back tracked branch tips, ANY companion-repo merge could turn host
`main` red with zero host commits (`ERR_PNPM_OUTDATED_LOCKFILE` across every
install-running job ‚Äî the incident #138 fixed, designed away by #141).

The prod image never had this problem: `docker build` acquires the bootable
set from the committed `cinatra-required-extensions.lock.json` (SHA-pinned
codeload tarballs, integrity-verified). The clone-back now matches that
discipline: **CI checks every companion repo out detached at a committed
sha**, so host CI is reproducible at every host commit, and companion drift
lands as a deliberate, reviewable bump PR ‚Äî never as spontaneous red.

## The two locks (a disjoint partition ‚Äî never duplicate a pin)

| Lock | Owns | Regenerated by |
| --- | --- | --- |
| `cinatra-required-extensions.lock.json` | the prod bootable set (`cinatra.requiredExtensions`; 8 packages == `cinatra.systemExtensions` ‚Äî the cover-gate floor, cinatra#151 Stage 4) ‚Äî also the image build's acquisition source | `scripts/extensions/update-required-extension-lock.mjs` |
| `cinatra-dev-extensions.lock.json` | every OTHER `cinatraDevExtensions` entry (63 packages) ‚Äî clone-back pins only (`packageName`, `repo`, `resolvedSha`) | `scripts/extensions/update-dev-extension-lock.mjs` |

The required lock stays the single authority for its packages; the dev lock
must never carry one of them (pinned sync and
`packages/extensions/src/__tests__/dev-extensions-lock.test.ts` both fail
closed on overlap, on a missing pin, and on a stale pin).

## Consumption

- **CI (all merge-gating jobs):** `.github/actions/clone-extensions` defaults
  to `mode: pinned` ‚Üí `node scripts/ci/sync-dev-extensions.mjs --pinned` ‚Üí
  detached checkout at each lock sha, verified (`HEAD == pin`, package.json
  `name` matches the declared key). Fail-closed: an unpinnable universe, a
  dirty tree, or an unreachable pin is a hard error ‚Äî never a silent fallback
  to tips.
- **Local dev (`make setup` / `cinatra setup`):** unchanged tip-tracking.
  Devs want tips; only CI consumes the pins.
- **`CINATRA_<NAME>_REPO_URL` overrides:** an override is only an *alternate
  remote* (fork/mirror) ‚Äî it must still serve the pinned sha, and the lock's
  repo slug is validated against the *committed* config URL, not the
  override. An override never unpins.

## The floating-HEAD canary (non-gating)

`.github/workflows/extension-head-canary.yml` runs daily (and on dispatch)
against the companion repos' TIPS: divergence report (HEAD vs pin, one
`::warning::` per moved repo), agent validation, extension README gate,
`pnpm install --frozen-lockfile`, and the generated-maps `--check`. A red
canary means "tips have drifted past the pins in a way that will need
attention at the next bump" ‚Äî it gates nothing.

## Bumping the pins (one deliberate PR)

Who/when: whoever lands a companion-repo rollout that needs host integration,
or whoever picks up a red/warning canary run; otherwise on demand. There is no
scheduled auto-bump PR (repo Actions policy does not allow workflow-created
PRs), so the canary is the cadence signal and a human lands the bump.

```sh
# 1. re-pin (all, or --select a,b for a targeted bump)
node scripts/extensions/update-dev-extension-lock.mjs
# (required-set packages instead use: node scripts/extensions/update-required-extension-lock.mjs --select x)

# 2. re-sync the local tree to the new pins
node scripts/ci/sync-dev-extensions.mjs --pinned

# 3. pnpm-lock.yaml moves WITH the pins (the frozen install validates against
#    the pinned manifests, so these two must land in the same commit)
pnpm install

# 4. the committed generated maps must match the pinned tree
node scripts/extensions/generate-extension-manifest.mjs
node scripts/extensions/generate-extension-manifest.mjs --check

# 5. commit lock(s) + pnpm-lock.yaml + any regenerated maps together; open the PR
```

The bump PR's own CI run *is* the integration test for the new pins: any
breakage lands on the bump PR, not on `main` and not on unrelated lane PRs.

## Failure modes

- **Unreachable pin** (companion force-push/GC): the pinned fetch fails loud
  ‚Üí regenerate that entry (`--select`) in a bump PR. Never weaken to
  tip-tracking to "fix" this.
- **Pin/lockfile skew** (lock bumped without `pnpm install`, or vice versa):
  the frozen install fails on the PR that introduced the skew ‚Äî keep them in
  one commit (recipe above).
